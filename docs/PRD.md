## 1. 产品定位

### 1.1 一句话定位

**为头条生态中长图文领域的个人创作者与中小团队,打造一个由 AI 全程陪伴的内容创作-审核-分发一体化工作台。**

### 1.2 三角色叙事(沿用 PDF 原话)

PDF 课题背景中,出题方亲自给平台定义了三个角色 —— 这三个角色就是本 PRD 的纲领,所有功能都围绕它们展开:

| 角色 | 含义 | 对应挑战 | 在产品里的表现 |
|---|---|---|---|
| **得力助手** | 辅助完成创意构思、素材整合、多模态内容生成 | 挑战一 | 创作工作台、双轨创作、富文本编辑器、作者私人 Prompt 体系、素材库 |
| **守门员** | 在创作各阶段自动识别并干预合规风险,评估质量 | 挑战二 | 5 阶段审核链路、4 维质量分、规则库、识别-干预-反馈闭环、素材合规校验、平台保留 Prompt 与 PE 工程化 |
| **导航员** | 让优质内容脱颖而出,提供数据反馈形成良性循环 | 挑战三 | 热点榜、爆文榜、信息流、数据回流到工作台 |

### 1.3 核心命题:人机协同

PDF 提示语中明确指出本课题是「**一场关于'人机协同'创作模式的深度思考**」。本 PRD 的任何一个 AI 触点,都遵循下面这条设计准则:

> **AI 提主张,人做决定。** AI 永远不直接修改用户的内容,只生成「候选」与「建议」;人始终是最终决策者。所有的 AI 介入都伴随明确的 Accept / Reject / Modify 三态选择。

进一步,作者对 AI 的掌控不止于"对每条输出 Accept/Reject":创作类 AI 工具背后的 Prompt **本身**也由作者维护 —— 作者决定 AI **如何思考**(详见 §3.5)。平台只在安全/评分等底线职责上保留对 Prompt 的独占控制。

唯一例外是「高危违规自动拦截」,这是平台底线、不接受人为覆盖。

### 1.4 目标用户画像

| 维度 | 描述 |
|---|---|
| **典型用户** | 头条领域作者,粉丝量 0–5w,有自己擅长的垂类(科技/财经/职场/历史/三农等),没有专业团队 |
| **创作频次** | 每周 2–5 篇,单篇 1500–3500 字,3–8 张配图 |
| **核心痛点** | 选题灵感断、不会写标题、配图找得累、发出去才知违规、看到数据低不知怎么改 |
| **不是谁** | **不是** 专业 MCN(不做矩阵管理)、**不是** 短视频达人(不做剪辑)、**不是** 小红书种草(不做卡片式编辑器) |

### 1.5 不做什么(明确剔除以聚焦)

- 不做视频剪辑、配音、卡点
- 不做小红书形态的多图卡片编辑器(本平台聚焦长文)
- 不做创作者矩阵账号管理
- 不做营收/广告分成结算
- **不做外部热点话题抓取**(PDF 中"热点追踪"加分项,超出 3 周交付范围,本平台"热点榜"指代平台内部内容热度榜)

## 2. 核心功能架构

```
                ┌──────────────────────────────────────┐
                │         三角色叙事的产品骨架          │
                └──────────────────────────────────────┘

  ┌─── 得力助手 ────┐    ┌─── 守门员 ────┐    ┌─── 导航员 ────┐
  │  创作工作台    │    │  审核与质量   │    │  分发与发现    │
  ├──────────────┤    ├──────────────┤    ├──────────────┤
  │ 双轨创作      │ ←→ │ 5 阶段审核    │ ←→ │ 双榜单         │
  │ 富文本编辑器  │    │ 分级响应      │    │ 信息流首页     │
  │ 流式 AI 输出  │    │ 4 维质量分    │    │ 内容详情页     │
  │ 草稿与离线    │    │ 规则库        │    │ 排序公式       │
  │ 版本历史      │    │ 素材合规      │    │ 数据回流       │
  │ 作者 Prompt   │    │ 平台保留      │    │                │
  │  两层(私人)  │    │  Prompt + PE  │    │                │
  │ 素材库        │    │ 反馈闭环      │    │                │
  └──────────────┘    └──────────────┘    └──────────────┘
         ↑                                          │
         └─── 数据反馈回流(良性循环) ─────────────┘

           ┌───────────── 横切关注点 ─────────────┐
           │ 用户中心 │ 通知中心 │ 工程化与交付  │
           └──────────────────────────────────────┘
```

**三个模块通过两条「闭环」相连**,这是本设计的灵魂:

1. **质量闭环**(守门员 ↔ 得力助手):质量分的每一个低分维度,都对应工作台里的一个 AI 改写工具。"读者体验维度低 → 一键重写标题"。
2. **数据闭环**(导航员 → 得力助手):分发后的阅读、互动、留存数据回流到作者的工作台首页,以「下一篇该怎么改」的形式给出建议,而不是冷冰冰的看板。

## 3. 模块一:创作工作台(得力助手)

> 对应 PDF **挑战一:构建 AIGC 内容创作的全链路体验**

### 3.1 双轨创作模式

进入「新建创作」页时,用户在两种模式中选择。两种模式的差别只在**起步阶段**,进入编辑器后能力完全一致。

#### 3.1.1 快速稿(AI 主导生成)

- **入口**:首页「新建创作」→ 选择「快速稿」
- **对应 PDF 进阶挑战 #1**:「**用户仅提供核心素材和大致思路,即可由 AI 自动生成结构完整、表达丰富的图文内容**」。本平台中长图文形态下,该挑战表现为「选题描述 → AI 大纲 → 流式分段生成」的完整链路。
- **流程**:
  1. **输入选题**:用户填写 1–3 句话的选题描述,如「最近 AI 编程工具很火,我想写一篇普通人怎么用 Cursor 上手编程」
  2. **AI 出大纲**:平台调用大模型生成 3–8 段大纲,用户可调整顺序、增删段落、修改任意段标题
  3. **逐段流式生成**:用户点击「生成全文」,AI 按大纲顺序流式输出每一段。生成过程中:
     - 用户可随时点「停下/重生成/继续」
     - 每段生成完毕后立即触发**生成阶段审核**(详见 4.1.3)
  4. **进入精修**:全文生成完毕,自动转入精耀稿模式继续编辑
- **AI 介入点**:选题理解、大纲生成、分段写作、配图建议
- **人决策点**:大纲调整、段落取舍、最终内容
- 对应 PDF 进阶挑战:「**用户仅提供核心素材和大致思路,即可由 AI 自动生成结构完整、表达丰富的图文内容**」

#### 3.1.2 精耀稿(人主导,AI 随需调用)

- **入口**:首页「新建创作」→ 选择「精耀稿」 / 已有草稿继续编辑 / 快速稿生成完毕后
- **形态**:富文本编辑器(详见 3.2),用户自由打字,AI 工具放在右侧工具栏,按需触发
- **AI 工具集**(全部按段落或选区作用,不修改原文,只生成候选;每个工具都绑定**当前生效 Prompt**,作者可基于平台内置 Prompt 复制一份到自己的私人层后编辑,详见 §3.5):
  | 工具 | 作用 | 对应质量分维度 | 当前生效 Prompt(默认) |
  |---|---|---|---|
  | 改写流畅 | 优化语病、长句拆短、口语化降级 | 表达质量 | 平台内置(默认款) |
  | 扩写补充 | 在事实/数据/案例方向延展 | 内容价值 | 平台内置(默认款) |
  | 转换文风 | 财经科普 / 故事化 / 深度评论 等 5 种预设 | 表达质量 | 平台内置(默认款) |
  | 起小标题 | 给当前段一个吸睛小标题 | 读者体验 | 平台内置(默认款) |
  | 起新标题 | 给整篇文章生成 5 个候选标题 | 读者体验 | 平台内置(默认款) |
  | 重写开头 | 强化前 3 句的钩子 | 读者体验 | 平台内置(默认款) |
  | 加数据/事实 | AI 检索补充可信数据(标注信息源) | 内容价值 | 平台内置(默认款) |
  | 加话题 | 推荐 3 个话题词 + 1 段引导互动的结尾 | 传播潜力 | 平台内置(默认款) |
  | 配图建议 | 根据正文推荐素材库中的图,或调起文生图 | 读者体验 | 平台内置(默认款) |

  > 默认状态下,所有工具都引用「平台内置 Prompt 库」的只读模板。作者可在「Prompt 管理」面板把任一内置 Prompt 复制到自己的私人层后编辑;**只要作者没复制,运行时直接读内置 Prompt;一旦复制并启用,运行时切换到该作者的私人副本**(详见 §3.5.2、§3.5.3)。

每个工具产出后,以「侧边对比卡」形式展示原文与候选,用户点击 Accept / Reject / 在候选基础上微调。**AI 不直接覆盖用户内容**,这是 1.3 节人机协同准则的体现。

### 3.2 富文本编辑器

#### 3.2.1 编辑器能力清单

| 能力 | 说明 |
|---|---|
| 基础排版 | 段落、加粗、斜体、引用、有序/无序列表、分隔线 |
| 标题层级 | H2 / H3 两级(限制三级以下,符合中长图文规范) |
| 媒体插入 | 图片(从素材库 / 上传 / AI 生图)、链接 |
| 块结构 | 段落级是最小操作单位,支持拖拽重排 |
| 流式插入 | 支持 AI 流式 token 边到达边渲染,无闪烁、无阻塞输入 |
| 协同友好 | 内部数据结构基于「快照 + 操作」,为版本历史与冲突解决留出空间 |

#### 3.2.2 流式 AI 输出渲染规则

- 流式输出区域以独立块呈现,**视觉上区别于已确认内容**(浅灰背景 + 流光指示)
- 流中允许用户继续在文档其他位置编辑,不阻塞键盘输入
- 流中点「停下」 → 当前已生成内容保留,转为可编辑状态
- 流中命中**生成阶段审核**违规 → 该段落变红色边框,弹出「修复建议 / 重新生成 / 仍要保留」三选一(详见 4.1.3)

### 3.3 内容生命周期状态机

PDF 同时要求:30 秒自动保存(挑战一)、断网继续(挑战一)、网络恢复同步(挑战一)、发布后二次编辑(挑战一)、下线/撤回/回滚(进阶挑战)。这些需求拼起来就是一条**完整的内容生命周期**,必须用状态机来定义,否则边界不清。

状态机共 **8 个状态**:本地草稿 / 本地未同步 / 云端草稿 / 待审核 / 审核驳回 / 已发布 / 已下线 / 已回滚。其中「审核驳回」是「待审核」的失败分支,在 UI 上仍归类为草稿态(可继续编辑)。

```
                    ┌──────────────────┐
   新建 ──────────→ │   本地草稿       │
                    │  (LocalDraft)    │
                    └────────┬─────────┘
            30秒自动保存 / 失联恢复
                             ↓
                    ┌──────────────────┐ ←── 断网时回退到这里
                    │  本地未同步      │
                    │  (DirtyLocal)    │
                    └────────┬─────────┘
                  网络恢复 / 立即点保存
                             ↓
                    ┌──────────────────┐
                    │   云端草稿       │ ──── 用户随时关闭 / 多设备打开
                    │  (CloudDraft)    │
                    └────────┬─────────┘
                       点击发布
                             ↓
                    ┌──────────────────┐
                    │   待审核         │ ←─── 阻塞在审核阶段
                    │  (Reviewing)     │
                    └─┬────────────┬───┘

                ↓            ↓
        ┌──────────────┐  ┌──────────────┐
        │   已发布     │  │  审核驳回    │
        │ (Published)  │  │  (Rejected)  │ → 回到 CloudDraft
        └──┬─────────┬─┘  └──────────────┘

   ┌──────────────┐
   │   已下线     │ → 作者可点「恢复」重新提审 → Reviewing
   │  (Offline)   │

          │  作者「回滚到某个版本」
          ↓
   ┌──────────────┐
   │   已回滚     │ → 当前生效版本变更,旧版进入版本历史归档
   │ (RolledBack) │
```

#### 3.3.1 草稿自动保存策略 [硬指标:30s]

- **本地保存**:每次有效编辑后 1 秒防抖,写入 IndexedDB(承担断网兜底)
- **云端保存**:每 30 秒触发一次,或用户主动 Cmd+S,以**增量 patch** 方式上传
- **保存状态指示**:右上角持续显示「已保存到云端 · 12:34:56」/「未保存(离线中)」/「同步中 …」
- 验证方式:在编辑器里手动断网,持续编辑 1 分钟,恢复网络后看到全部内容已同步;并且**网络中断时无任何报错弹窗或编辑中断**。

#### 3.3.2 断网与多设备冲突解决

- **离线编辑**:断网时编辑器照常工作,所有修改写入 IndexedDB,UI 顶部出现一条信息条「当前离线,内容已保存在本设备」
- **网络恢复**:平台对比本地与云端版本号:
  - 本地较新 / 等同 → 直接 patch 上传
  - 云端较新(其他设备改过) → **以云端为准刷新编辑器**,本设备的离线改动以「冲突备份」形式单独保留在版本历史中,作者可手动从备份取片段
- **多设备并发**:同一篇文章被多端打开时,后打开的端弹出「该文章在其他设备打开,是否进入只读模式」

#### 3.3.3 发布后二次编辑

- 已发布内容点「编辑」 → 状态切回 CloudDraft,但**线上版本仍保留原内容**;新版本编辑完保存提交后再次进入 Reviewing
- 审核通过后,新版本**替换**线上展示;**保留旧版本作为历史版本**(版本历史功能可见)
- 二次编辑统一**重过完整发布前审核(§4.1.4)**;新版本上线后,**热度是否重置/继承**由后台开关控制(默认继承,可由运营按品类/事件切到重置)

#### 3.3.4 下线/撤回/回滚(进阶)

- **作者主动下线**:已发布 → 已下线,前台不可见,后台可恢复
- **平台强制下线**:发布后审核(详见 4.1.5)抽查命中、用户举报核实、规则库更新批量复审命中 → 平台强制下线 + 通知作者 + 给出原因
- **回滚**:版本历史里任意旧版本可一键设为当前线上版本(需重新过审)

### 3.4 版本历史

- 每次云端保存生成一个版本节点;**单文档保留最近 30 个版本 + 所有「已发布过」的版本**
- 时间轴展示所有版本,每个节点显示时间、字数、与前一版本的 diff 概览
- 任意两个版本可点选进入「左右对照 diff 视图」
- 版本节点支持「设为当前 / 恢复为草稿」

> 设计取舍:版本历史不是单独开发的功能,而是从 3.3.1 草稿快照机制中**自然导出**的产品化呈现。极小的设计代价换来评委可一眼感知的产品氛围。

### 3.5 Prompt 体系:作者主导,平台守底

PDF 交付物清单要求「平台效果评估与内容分发优化报告」中包含「**大模型使用的时候,PE 多次调教评估过程**」。但「PE」在本平台不是少数运营工程师 —— **作者本人就是创作类 AI 工具的 PE**。本平台采用「**作者主导,平台守底**」的 Prompt 体系:作者完全掌控自己创作类 AI 工具背后的 Prompt,平台只在两端守住边界 —— 提供高质量「内置 Prompt」作为起步参考、保留安全/评分类 Prompt 作为不可逾越的底线。

这是 §1.3 人机协同准则的延伸:作者不止决定 AI 的每一次输出,还掌控 AI **如何思考**。

#### 3.5.1 两层架构

| 层级 | 拥有者 | 谁可写 | 谁可读 | 是否会真正生效 |
|---|---|---|---|---|
| **平台内置 Prompt 库** | 平台 | 平台 PE 工程师 | 全体作者(含设计注释) | ✅ 作者未自定义时直接生效(默认款) |
| **作者私人层** | 作者本人 | 仅作者本人 | 仅作者本人 | ✅ 作者基于内置 Prompt 复制并启用后,替代该作者的内置默认生效 |

> **核心约束**:任何真正在 AI 工具上跑的 Prompt,要么是平台内置 Prompt 默认款(只读),要么是该作者的私人副本。**没有第三方写入路径**,这是「作者主导」的物理保证。

#### 3.5.2 Prompt 与 AI 工具的绑定

§3.1.2 列出的 9 个**创作类 AI 工具**(改写流畅、扩写补充、转换文风、起小标题、起新标题、重写开头、加数据/事实、加话题、配图建议)各自有一个**当前生效 Prompt**:

- 默认状态:运行时直接读取**平台内置 Prompt 库**的默认款 —— 对作者透明,体验等同开箱即用
- 作者在「Prompt 管理」面板对某个工具点「自定义」:把当前内置 Prompt 复制一份到自己的私人层,跳到编辑页;保存并启用后,该工具切换到读取该作者的私人副本
- 作者可随时点「恢复默认」,工具退回到读取内置默认款

#### 3.5.3 作者主操作流

```
[ 浏览内置 Prompt 库(只读) ]
        │
        ▼   "自定义" → 复制到私人层
[ 副本进入"我的 Prompt" ]
        │
        ▼   编辑 system prompt / 参数 / few-shot,保存
[ 启用为某 AI 工具的"当前生效 Prompt" ]
        │
        ▼  随时
[ 恢复默认(回到平台内置 Prompt) ]
```

- **极简版本管理**:每条作者私人 Prompt 保存即覆盖,只保留最近 3 个历史快照,可点击回滚 —— 这是 PDF 「PE 多次调教评估过程」在作者侧最朴素的表达,不做沙盒、不做 A/B 对照,把这些复杂能力留给平台保留 Prompt(§4.7.3)
- **效果反馈**:作者侧的「PE 调教效果」体现在 §6.5 报告里 —— 启用了私人 Prompt 的工具调用量、Accept 率与默认款的对比

#### 3.5.4 平台内置 Prompt 库(精选示范)

- **数量克制**:每个 AI 工具配 1 个默认款 + 1–2 个风格款,**全平台合计 ≤ 30 条内置 Prompt**,贵在精不在多
- **每个内置 Prompt 包含**:system prompt、推荐参数、3 条 few-shot 示例、**1 段「设计注释」**(平台 PE 解释为什么这么写、规避了什么坑、适合什么品类)—— 这段注释是内置 Prompt 库最重要的资产,把 PE 经验显式传递给作者
- **迭代节奏**:内置 Prompt 库每月小迭代,新增/修订的依据来自 §6.5 报告里的工具调用数据(低 Accept 率工具优先重写),走 §4.7.3 实验室流程

> **本节边界**:§3.5 只覆盖**作者侧的创作类 Prompt**(改写、扩写、起标题等 9 个工具)。平台用于安全审核、质量评分、配图诊断、数据回流诊断的 Prompt **不在本节范围内**,它们由平台独占维护、对作者不可见,详见 **§4.7 平台保留 Prompt 与 PE 工程化**。这条边界对应 §1.3 准则的延伸:**作者塑造 AI 提主张的方式;平台塑造 AI 把关的方式**。

### 3.6 素材库

PDF 原话:「**多媒体素材管理**」并「**对上传的素材进行基础的合规性校验**」。本平台中长图文场景下,素材以**图片**为主。

#### 3.6.1 素材入库三种来源

| 来源 | 处理 |
|---|---|
| **上传** | 上传时即时审核(图像合规、版权水印检测),通过后入库 |
| **AI 生图** | 调用文生图模型,提示词与生成结果都入库,带「AI 生成」水印标注 |
| **从开放图库引入** | 集成 1–2 个版权友好的开放图库 API(如 Unsplash),支持搜索后引入 |

#### 3.6.2 素材自动打标签

每张图入库时调用多模态模型,自动产出:
- **场景标签**:办公室、街道、咖啡馆…
- **主体标签**:人物、动物、产品…

作者搜索素材时支持按标签过滤;作者编辑时,系统根据正文段落语义**主动推荐**素材库中的图。

> 素材**入库时**与**插入文章前**的合规校验(肖像权、版权水印、敏感内容、AI 未标注)由守门员负责,详见 **§4.6 素材合规校验**。

## 4. 模块二:安全与质量管控(守门员)

> 对应 PDF **挑战二:打造 AI 驱动的内容安全与质量「防火墙」**

PDF 评分维度第四条「内容安全与质量闭环(10%)」明确要求:**「识别-干预-反馈」闭环的完整性**。本模块的全部设计就是这条闭环的展开。

### 4.1 5 阶段审核链路

PDF 原话「**在创作的各个阶段** 自动识别并干预潜在的合规风险」。本设计把审核拆到创作生命周期的 5 个时刻:

| 阶段 | 时机 | 形式 | 主要技术 |
|---|---|---|---|
| **① Prompt 阶段** | 用户填选题/提示词,失焦时 | 弹窗:「该选题可能涉及 [医疗夸大/政治敏感],建议调整方向」 | 规则库 + 轻量分类模型 |
| **② 输入阶段** | 用户键入,1.5s 防抖 | 编辑器内违规词高亮(下划红线),不阻塞输入 | 本地敏感词库(数千词级) |
| **③ 生成阶段** | AI 流式生成时 | 段落级实时审核,违规段红框 + 修复建议 | 大模型 + 规则 |
| **④ 发布前阶段** | 用户点「发布」 | 全文综合审核,展示「安全分 + 质量分」+ 处置建议 | 大模型多维评估 |
| **⑤ 发布后阶段** | 内容已上线 | 用户举报触发 / 抽样巡检 / 规则更新批量复审 | 异步审核管道 |

每个阶段都对应**人机协同准则**:平台只识别和提示,作者决定怎么处理;只有「高危」**强制拦截**。

#### 4.1.1 Prompt 阶段审核

- 触发:用户在快速稿入口写选题、或个人 prompt 输入完成
- 检测:涉政、涉医疗夸大宣传、涉金融诱导、涉违法导向等高危选题方向
- 响应:弹窗提示具体风险类目,作者可「换个角度」/「我有把握继续」
- 命中数据回流到规则库迭代

#### 4.1.2 输入阶段审核

- 触发:作者在编辑器中打字,本地 1.5 秒防抖
- 检测:本地敏感词库(数千级,主要覆盖政治、低俗、广告引流)
- 响应:命中词下划红色波浪线,鼠标悬停显示提示,**不打断输入节奏**
- 性能要求:本地词库扫描 P99 < 50ms,绝不阻塞主线程

#### 4.1.3 生成阶段审核(流式)

- 触发:AI 流式生成时,**每生成一个段落**就送审一次
- 检测维度:涉黄、涉赌、涉毒、涉政敏感、低俗、虚假宣传
- 响应:
  - 段落违规 → 该段落红框 + 弹出小卡:**「重新生成 / 修改建议 / 仍要保留」**
  - 多段连续违规 → 中断流式生成,提示作者「当前选题方向风险较高,建议调整」
- 注意:这是和 PDF 「**在创作的各个阶段干预**」最直接对应的设计 —— 不等到发布才审,在生成过程中就介入

#### 4.1.4 发布前阶段审核

- 触发:作者点「发布」按钮
- 检测:**全文级别**多维度审核 + 4 维质量分(详见 4.3)
- 响应:模态弹窗,左侧文章预览,右侧:
  ```
  安全分: 92 / 100   ✅ 通过
   ├─ 涉黄        0   通过
   ├─ 涉赌        0   通过
   ├─ 涉毒        0   通过
   ├─ 政治敏感    0   通过
   ├─ 低俗内容    8   轻微 (1 处, 见正文标注)
   └─ 虚假宣传    0   通过
  
  质量分: 76 / 100   ⚠ 良好,可优化
   ├─ 内容价值    82
   ├─ 表达质量    78
   ├─ 读者体验    65   ← 标题钩子较弱,建议优化
   └─ 传播潜力    79
  
  [立即发布]  [先优化再发]
  ```
- 命中**高危**:不展示「立即发布」按钮,只能「修改后重提」
- 命中**中危**:可发布,但显眼提示
- 全部低危:可直接发布

#### 4.1.5 发布后阶段审核

三种触发路径:
1. **用户举报**:每条已发布内容右上角有「举报」入口,举报理由分类化 → 后台审核员复审 → 高危下架
2. **平台抽样巡检**:每天对发布量按 5% 抽样,异步走全文大模型审核
3. **规则更新批量复审**:规则库新增/调整规则后,对历史已发布内容增量复审命中部分

每次发布后审核命中,平台会:
- 对内容采取「自动下线 / 强制修改 / 警告」
- 给作者发系统通知,说明原因和参考的具体规则条目
- 命中数据进入**反馈闭环**(4.4)

### 4.2 分级响应策略

PDF 原文:「自动拦截 **或** 一键生成合规替代内容」。这里 OR 关系实际是按严重等级分的:

| 等级 | 类目示例 | 响应 |
|---|---|---|
| **高危** | 涉政、涉黄、涉毒、涉赌、煽动暴力 | **强制拦截**,不可发布;已发布则强制下线 |
| **中危** | 标题党、夸大宣传、低俗暗示、医疗误导 | 弹「**一键生成合规替代**」选项,作者决定接受/修改/坚持 |
| **低危** | 用词不当、错别字、表述模糊、AI 痕迹明显 | 侧栏温和提示,不阻断流程 |

「一键生成合规替代」本质是调用大模型,以「保留原意但去除违规元素」为约束重写违规段落,产出 2 个候选给作者选。

### 4.3 4 维质量分体系

PDF 原话:「**预先定义的 AI 文章好坏标准 …… 多维度评判**」。本平台采用 4 维:

| 维度 | 评估的子项 | 对应工作台 AI 工具 |
|---|---|---|
| **内容价值** | 信息增量、原创性、专业准确性、数据支撑 | 扩写补充、加数据/事实 |
| **表达质量** | 语言流畅、逻辑结构、文风统一、句式多样 | 改写流畅、转换文风 |
| **读者体验** | 标题吸引力、开头钩子、小标题清晰、配图相关 | 起新标题、起小标题、重写开头、配图建议 |
| **传播潜力** | 话题热度、情感共鸣、互动引导、时效性 | 加话题 |

**重要的设计**:每个维度低分都对应一个具体的 AI 工具。在 4.1.4 发布前审核弹窗里,作者点击「读者体验 65」 → 自动跳到工作台,带上这个维度作为上下文,主动建议作者调用「起新标题」工具。这就是 1 节里说的**质量闭环**。

#### 4.3.1 评分实施

- 每个维度由大模型打 0–100 分,**附带 1–2 句解释**(为什么扣分)
- **单次评分,结果缓存**到内容元数据中,供分发排序与作者参考(避免重复消耗大模型 token)
- 评分稳定性的衡量放到**离线评估**:从历史内容随机抽样,对同一篇做 3–5 次评分,统计标准差,作为 §6.5 报告里的「评分稳定性」指标

#### 4.3.2 质量分作为分发依据

PDF 原话:「**作为后续分发排序的重要依据**」。在挑战三的排序公式(5.4)中,质量分是**第一权重**(α=0.5)。

### 4.4 审核规则库

PDF 交付物清单第 4 项:**「内容安全审核规则库与质量评估体系说明」**,且要求**详细定义规则边界**。

#### 4.4.1 规则库结构

```yaml
rule_id: SEC-POLITICS-001
category: 涉政敏感
severity: high            # high / medium / low
description: "对国家领导人的负面或调侃言论"
keywords: [...]            # 触发关键词(可选)
prompt_template: "..."     # 大模型审核 prompt 模板
examples_positive: [...]   # 正样本(应被命中)
examples_negative: [...]   # 负样本(应被放过)
status: active             # active / deprecated
version: 1.2
created_at, updated_at
```

#### 4.4.2 规则库管理

- 规则按 category 树状组织(政治 / 色情 / 赌博 / 毒品 / 低俗 / 虚假 / 版权 / 其他)
- 每条规则附带**正负样本测试集**,新增/修改时自动跑测试集校验
- 每条规则可独立**启用/停用**
- 平台运营端有规则库管理 UI(评委可查看)

#### 4.4.3 准确率达标 [硬指标:≥ 90%]

- 维护一个 **300 条标注样本**的离线测试集(每个高危类目至少 30 条)
- 每次规则变更后,**自动跑测试集 → 准确率不达标不允许上线**
- 每周生成准确率报告,纳入交付物中

### 4.5 识别-干预-反馈闭环

```
        识别                干预                  反馈
   ┌──────────┐       ┌──────────┐       ┌──────────────┐
   │ 5 阶段   │ ───→  │ 分级响应 │ ───→  │ 审核日志     │
   │ 审核链路 │       │  + 一键   │       │ 误判反馈     │
   │          │       │  替代     │       │ 准确率监控   │
   └──────────┘       └──────────┘       └──────┬───────┘
        ↑                                          │
        │                                          ↓
        └────── 规则库迭代 ←─── 周报 + 灰度 ────┘
```

**反馈环节**有两个反馈源:
1. **作者反馈**:被拦截/改写后,作者可点「我觉得不该被拦」→ 进入**误判队列**
2. **质量监控**:测试集准确率低于阈值时报警

**反馈数据**用于:
- 高频误判的规则进入审查队列,人工评估后修正
- 高频漏判的内容类型(用户举报命中的类型)进入新规则起草
- 规则库每周一次小迭代,每月一次大版本

### 4.6 素材合规校验

PDF 原话:「**对上传的素材进行基础的合规性校验**」。素材合规与正文的安全审核(§4.1)同属守门员职责,但作用对象与触发时机不同,故单列一节。

#### 4.6.1 触发时机(两次校验)

| 时机 | 校验目的 | 失败处理 |
|---|---|---|
| **入库时** | 阻止违规素材进入素材库,污染源头 | 拦截:不入库,告知作者原因 |
| **插入文章前** | 即便已在库,插入正文时再过一遍最新规则(规则库可能已更新) | 警告:作者可选择"换图 / 仍使用",仍使用时记录到 §4.1.4 发布前审核证据链 |

#### 4.6.2 校验维度

- 是否含**真实人脸**(涉及肖像权风险)
- 是否含**版权水印 / logo**
- 是否含**敏感图像内容**(暴力、低俗)
- 是否为**AI 生成且未标注**(平台规则)

校验结果分**通过 / 警告 / 拦截**三档,与 §4.2 分级响应策略对齐。底层调用的 Prompt 见 §4.7.1「配图诊断 Prompt」。

### 4.7 平台保留 Prompt 与 PE 工程化

守门员的 AI 把关能力,本质上也是大模型驱动的 —— 但与 §3.5 作者侧的创作类 Prompt 不同,**这一类 Prompt 必须由平台独占**:它们承担安全红线、质量评分、合规诊断等公信力职责,**任何作者侧的修改都可能动摇平台底线**。本节集中说明这套独占机制,并落实 PDF 交付物 #3 要求的「**大模型使用的时候,PE 多次调教评估过程**」。

#### 4.7.1 平台保留 Prompt 清单

| Prompt 类型 | 用途 | 服务章节 |
|---|---|---|
| 安全审核 Prompt | 5 阶段审核中所有大模型环节(Prompt / 生成 / 发布前 / 发布后) | §4.1 |
| 4 维质量评分 Prompt | 内容价值 / 表达质量 / 读者体验 / 传播潜力打分 | §4.3 |
| 配图诊断 Prompt | 肖像权 / 版权水印 / 敏感图像 / AI 未标注识别 | §4.6 |
| 素材打标签 Prompt | 场景 / 主体多模态识别 | §3.6.2 |
| 数据回流诊断 Prompt | 「我的创作」页的行动建议生成 | §5.5 |

#### 4.7.2 与 §3.5 作者私人 Prompt 的边界

| 维度 | §3.5 作者私人 Prompt | §4.7 平台保留 Prompt |
|---|---|---|
| 谁来写 | 平台 PE(内置)/ 作者(私人副本) | 仅平台 PE |
| 谁能改 | 作者(在自己的私人层) | 仅平台 PE,作者**不可见、不可修改** |
| 在哪生效 | 作者侧创作工具(改写、扩写、起标题等 9 个) | 平台侧把关链路(审核、评分、诊断、打标签) |
| 失败影响面 | 作者本人单篇质量 | 平台公信力、安全合规、生态秩序 |

这条边界是 §1.3 人机协同准则的延伸:**作者塑造 AI 提主张的方式;平台塑造 AI 把关的方式**。两套权限体系**物理隔离**,互不串扰。

#### 4.7.3 PE 工程化:Prompt 实验室

平台保留 Prompt 的每次更新,都通过 Prompt 实验室走一遍标准化流程,产出可审计的迭代证据(交付物 #3 报告的核心数据来自这里):

1. **测试集**:每个 Prompt 类型维护一个脱敏评估集(安全 ≥ 300 条覆盖 PDF 所列 7 大类目;质量评分 ≥ 100 条人工标注;诊断类 ≥ 50 条)
2. **批量评估**:新版 Prompt 在测试集上批量跑,自动出指标 —— 安全准确率(对应 [硬指标:≥ 90%])、评分稳定性(同文重复打分方差)、人工抽样满意度
3. **版本对比**:与上一版逐项对比,关键指标不回退、目标指标提升才允许上线
4. **人工确认上线**:运营同学在管理 UI 看到对比报告与样例,确认后**一键切换为当前版本**;**保留上一版作为「快速回滚」选项**,出问题手动一键回退
5. **可追溯**:全部历史版本、测试集、指标曲线、回滚记录持久化存储,以飞书在线表单形式作为交付物 #3 报告的附录

> 与 §3.5.4 平台内置 Prompt 库迭代的关系:内置 Prompt 库迭代也走相同的实验室流程,但由于内置 Prompt 不直接影响"在线安全/评分",评估侧重点是「平均质量分提升」与「Accept 率」,不强卡 ≥ 90% 安全准确率红线。

### 5.2 信息流首页

- **作者其他作品**:页面底部推荐同作者最近 3 篇

### 5.4 排序公式(纯加权)

PDF 原话:「**依据内容质量分、阅读热度、发布时间等多维度进行综合排序**」

```
score = α · QualityScore  + β · HotnessScore  + γ · TimeDecayScore
```

| 项 | 含义 | 范围 | 默认权重 |
|---|---|---|---|
| QualityScore | 4 维质量分总分 | 0–100 | α = 0.5 |
| HotnessScore | 近期阅读+互动的 min-max 归一化(详见下) | 0–100 | β = 0.3 |
| TimeDecayScore | `100 · e^(-Δt / τ)` | 0–100 | γ = 0.2 |

**HotnessScore 计算细节**:

- **统计窗口与 τ 一致**:在热点榜中统计**最近 12 小时**的指标,在爆文榜中统计**最近 72 小时**的指标(避免长期高量内容稀释短期热度)
- **原始热度** `raw = 1·impression + 3·click + 5·dwell_unit + 8·like + 10·collect + 12·share - 30·report`,其中 `dwell_unit` 为**完读次数**(dwell ≥ 文章预估阅读时长 80%)
- **归一化**:对当前榜单候选池(同窗口同榜单)做 min-max 归一化到 0–100,即 `HotnessScore = 100 · (raw - min) / (max - min + ε)`
- **小池兜底**:候选池 < 50 篇时使用历史 P95 作为 max,避免噪声放大

**两榜单的差异在 τ 与窗口**:
- **热点榜**:τ = 12 小时,统计窗口 12 小时(快衰减,内容只在热点期上榜)
- **爆文榜**:τ = 72 小时,统计窗口 72 小时(慢衰减,经典内容能续航)

**进阶**:权重在运营管理 UI 中可配置,默认 α=0.5 / β=0.3 / γ=0.2。

#### 5.4.1 数据埋点清单(供 HotnessScore 与数据回流使用)

| 事件 | 触发时机 | 关键字段 |
|---|---|---|
| `impression` | 卡片进入视口 ≥ 500ms | post_id, user_id, position, source(rank/feed) |
| `click` | 用户点开详情 | post_id, from(feed/rank/profile) |
| `dwell` | 详情页停留时长(秒级心跳上报,离开时定结) | post_id, dwell_seconds |
| `read_complete` | 滚动至文末 ≥ 90% **且** dwell ≥ 预估阅读时长 80% | post_id |
| `like` / `collect` / `share` | 用户点击 | post_id |
| `report` | 用户举报(同时触发 §4.1.5) | post_id, reason |

**完读率(数据回流诊断使用)** = `read_complete / click`,**互动率** = `(like + collect + share) / click`。这两个指标支撑 §5.5 中的诊断规则(如「阅读量高、完读率低 → 重写开头」)。

### 5.5 数据反馈回流(良性循环)

PDF 原文:「**为创作者提供数据反馈,形成一个从创作、审核、分发到优化的良性循环**」

「我的创作」页**不是单独的数据看板**,而是把作者的下一步行动建议化:

| 一篇文章发布 24h 后 | 作者会看到 | 行动建议 |
|---|---|---|
| 阅读量低、质量分高 | 「**这是一篇好文章,但被埋了**」 | 推荐改换标题再次曝光(标题 A/B)|
| 阅读量高、完读率低 | 「**标题吸引,但留不住**」 | 推荐用「重写开头」工具优化前 3 句 |
| 阅读量低、完读率高 | 「**写得好,但话题冷**」 | 推荐用「加话题」工具加入热门话题词 |
| 互动率低 | 「**缺少互动钩子**」 | 推荐用「加话题」工具补结尾互动引导 |

**这就是质量闭环和数据闭环的合流**:数据 → 诊断 → 推荐工具 → 用户回到工作台改写 → 重新发布。

## 8. 风险与取舍

| 风险 | 应对 |
|---|---|
| 大模型 API 不稳定/限流 | 关键调用做 fallback(多 provider)、缓存高频结果、超时降级 |

| 进阶挑战时间不足 | 双轨创作的「快速稿」、智能榜单的「动态权重」、发布后审核的「批量复审」依次为可砍项 |
